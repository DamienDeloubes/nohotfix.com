Create a new database migration for: $ARGUMENTS

## Pre-Flight

1. **Load database context** — Read `.claude/skills/database.md` to understand the full schema, relationships, conventions, and JSONB structures
2. **Read current schema** — Read `packages/db/src/schema.ts` to see the current Kysely types
3. **Check existing migrations** — List files in `packages/db/src/migrations/` to determine the next migration number

## Impact Assessment

Before writing any code, assess and report:

1. **Tables affected** — Which tables are created, altered, or have new relationships?
2. **Tenant isolation** — Does any new table need `org_id`? (Answer: yes, unless it's `users` or a cross-tenant lookup table)
3. **Downstream changes** — List all files that will need updating:
   - `packages/db/src/schema.ts` — table interface changes
   - `packages/db/src/index.ts` — new type re-exports
   - `apps/api/src/adapters/repositories/` — affected repository implementations
   - `packages/domains/*/src/entities/` — affected domain entities
   - `packages/domains/*/src/ports/` — affected port interfaces
   - `packages/shared/src/schemas/` — affected Zod schemas
4. **Snapshot impact** — If touching authoring tables (spec_library, playbooks, playbook_sections, playbook_specs), does the snapshot service need updating?
5. **State machine impact** — If adding status columns or new CHECK constraints, do they align with the existing state machines?
6. **JSONB structure** — If adding JSONB columns, document the expected structure

Present the impact assessment to the user and wait for confirmation before proceeding.

## Migration File

3. Create `packages/db/src/migrations/{NNN}_{description}.ts` with both `up()` and `down()` functions
4. Use raw SQL via the `sql` template tag from Kysely (prefer raw SQL over Kysely schema builder for DDL)
5. Use `IF EXISTS` / `IF NOT EXISTS` for idempotency
6. When adding columns to existing tables: add nullable first → backfill if needed → set NOT NULL → add indexes
7. Name CHECK constraints as `{table}_{column}_check` for predictable drops in `down()`

## Schema & Exports

8. Update `packages/db/src/schema.ts` to reflect the new table structure:
   - Use `Generated<string>` for auto-UUID PKs
   - Use `ColumnType<Date, string | undefined, never>` for `created_at`
   - Use `ColumnType<Date, string | undefined, string>` for `updated_at`
   - Use `unknown` for JSONB columns
   - Add `Selectable<T>`, `Insertable<T>`, and `Updateable<T>` (only if updates are expected) convenience aliases
9. Re-export any new types from `packages/db/src/index.ts`

## Verification

10. Run `pnpm --filter @nohotfix/db build` to verify the migration compiles

## Rules

- snake_case for all column and table names
- Every tenant table MUST have an `org_id` column
- UUIDs for all primary keys (auto-generated by `uuid_generate_v4()`)
- TIMESTAMPTZ for all date columns (never plain TIMESTAMP)
- TEXT with CHECK constraints for enums (not PG enums)
- JSONB for rich text, test steps, artifact requirements, table data
- ON DELETE CASCADE only on parent-child within same bounded context
