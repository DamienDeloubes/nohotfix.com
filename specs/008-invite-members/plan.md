# Implementation Plan: Invite Members

**Branch**: `008-invite-members` | **Date**: 2026-03-08 | **Spec**: [spec.md](./spec.md)
**Input**: Feature specification from `/specs/008-invite-members/spec.md`

## Summary

Owners and Admins can invite users by email to join their organisation. Invites are stored in a new `invites` table (separate from `memberships`). On acceptance via a signed invite link, the invitee authenticates through WorkOS and a membership row is created. The feature includes resend (with token regeneration and rate limiting) and revoke capabilities, plus error screens for expired/revoked/already-accepted links.

**Primary context**: Identity
**Key technical decisions**: Separate `invites` table, crypto.randomBytes token, invite acceptance through `apps/web` with WorkOS auth redirect, `EmailPort` in domain for hexagonal email sending.

## Technical Context

**Language/Version**: TypeScript 5.7, Node.js 20
**Primary Dependencies**: Fastify 5, Kysely, TanStack Router + Query, React, Zod, WorkOS SDK, Resend SDK
**Storage**: PostgreSQL (new `invites` table via migration)
**Testing**: Vitest (unit + integration), Playwright (E2E)
**Target Platform**: Web (Linux server for API, Vercel for web/app)
**Project Type**: Web service (monorepo: API + web app + SPA)
**Performance Goals**: API read <500ms p95, API write <1000ms p95 (per constitution)
**Constraints**: org_id on all tenant queries, hexagonal architecture, no cross-domain imports
**Scale/Scope**: Low volume — invite operations bounded by seat limits (1–40 per org)
